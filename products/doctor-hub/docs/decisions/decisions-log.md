### D-192 — Agendamento resiliente: solicitação local + confirmação assíncrona; a realidade do CLIENTE é a fonte da verdade (2026-07-10)

Hoje o `POST /agendamentos` grava como fato consumado, **não valida se a vaga existe/está livre** e nunca envia ao cliente (`EnviadoTc=false` sempre — o **DEP-TC-1**). Isso torna o DoctorHub resiliente (agenda-se independentemente da disponibilidade do cliente) e resolve de quebra o furo de **duplo-agendamento** que o modelo síncrono tinha (dois operadores no mesmo slot → ambos viram solicitação, o cliente arbitra).
**Regra dura (Alessandro): "sempre assumimos a realidade do cliente".** O **sistema do cliente é SEMPRE a fonte da verdade** da vaga/agenda. Em conflito (vaga já ocupada lá), a solicitação volta **REJEITADA para nós** — o operador reage do nosso lado (escolhe outra vaga). **DESCARTADA** a ideia levantada antes de um **toggle/feature de "sobrepor no cliente"**: sobrepor significaria sobrescrever agenda alheia (risco de dois pacientes reais no mesmo horário) e violaria a **D-069** (sync com escrita nunca é destrutivo; `UPDATE` só com `WHERE` por `external_id`). Fonte-da-verdade-é-o-cliente não é configurável por cliente — é **regra da casa**.
**Consequência prática (a detalhar na spec):** `Agendamento` ganha **status/state machine** (`PENDENTE → ENVIANDO → CONFIRMADO | REJEITADO(conflito) | FALHA_TEMP(retry)`); a entrega ao cliente é **assíncrona e idempotente** (chave = id do agendamento, pra retry não duplicar); arquitetura recomendada = **Transactional Outbox no Postgres** (grava agendamento + evento na MESMA transação, mata o dual-write) + **entregador com retry/backoff** (no GCP, **Cloud Tasks** é o encaixe natural — Kafka/Rabbit são overkill pra este volume/contexto). Conecta com **DEP-TC-1** (esta é a integração real que estava pendente), **D-069** (o PUSH ao cliente segue o baseline: credencial dedicada, nunca destrutivo) e o **EMPI/D-191** (o `PacienteId` resolvido entra no pacote de sync). **Perguntas abertas** (não decididas — vão pra spec + discovery): o que o operador vê/faz no REJEITADO; quem pode agendar (D-011: Gestor ou Operador); reconciliação com o SALDO (D-190); formato/credencial do pacote de PUSH ao cliente. _(Estas foram respondidas em D-193/D-194.)_

### D-196 — Contrato dos endpoints pull+ack (auth, escopo, ack, LGPD, idempotência) (2026-07-11)
Alessandro decidiu os 5 pontos de segurança abertos em D-195, destravando a construção da Fase 2 (endpoints que o cliente consome). São **invariantes de segurança** — revisadas por humano, não delegadas cegamente.
1. **Auth do cliente = Keycloak client credentials (default) + API key dedicada como fallback.** Cada cliente que integra vira um **service account (confidential client)** no realm, com o `clienteId` no claim; faz `client_credentials` → JWT → validamos igual ao resto. Máquina-a-máquina (sem contexto de usuário). Para cliente **legado que não faça OAuth**, uma **API key dedicada por cliente (guardada como HASH**, nunca em claro) é o fallback. **Provisionar a credencial real de um cliente é operação de humano (onboarding)** — não se cria service account/API key de produção automaticamente (segredo/credencial = humano decide, baseline).
2. **Escopo (invariante dura):** o cliente **só enxerga as solicitações do próprio `clienteId`** (do token/credencial), **fail-closed** se ausente. Nunca um cliente vê o de outro.
3. **Ack = UMA fase.** O **pull é o "peguei"**: ao buscar, os itens retornados ficam `Enviando` com um **lease/visibility timeout**. O **confirm** resolve o resultado: `sucesso`→**Confirmado** | `conflito`→**Rejeitado** (cliente é a fonte da verdade — D-192). Lease expirado → volta a `Pendente` para novo pull. Confirm **idempotente por id do agendamento** (confirmar já-confirmado = no-op).
4. **LGPD/payload — identifica o paciente pelo id que o CLIENTE já tem:** manda o **`external_id`** (o `ExternalPatientId` do EMPI/D-191 — o id do paciente na base DELE) + ids de correlação nossos (id do agendamento, `PacienteId` do golden record) + dados do slot (vaga, especialidade, médico, unidade, horário). **SEM CPF, SEM nome completo**; iniciais dispensáveis. Não vazamos nada — o cliente já conhece essa pessoa. **Agendamento sem `external_id` resolvido NÃO sai por esse canal** (tratado depois; expor CPF seria decisão LGPD à parte, não tomada).
5. **Idempotência do pull = lease/visibility** (estilo fila): itens buscados ficam escondidos das buscas seguintes até confirmar ou o lease expirar — dois workers/retry do cliente nunca pegam o mesmo item em duplicado.
**Implementação (não é regra de negócio, anotar):** para filtrar por tenant é preciso **carimbar o `clienteId` no evento do outbox/agendamento** na criação — hoje o `Agendamento` não carrega `clienteId` explícito (deriva de unidade/solicitação); resolver a origem do `clienteId` no POST é pré-requisito técnico do pull. O worker/feature de integração fica **off por padrão** até o onboarding real. _(Este pré-requisito virou a fase própria D-197.)_

**Isolamento (refina D-198) por CONTENÇÃO de escopo:** o agendamento tem um **dono** na hierarquia (D-197 — via unidade/cliente que assumiu). O médico **vê** o agendamento se **algum vínculo ativo dele COBRE o dono** — vínculo no `Cliente` cobre todas as `Unidades` dele; vínculo só numa `Unidade` cobre só ela; vínculo no `Tenant` cobre tudo abaixo. Nunca vê fora dos seus vínculos.
**FORK para a discovery (não decidir agora):** vínculo em nível mais alto **cascateia** pros de baixo (recomendo: **sim, com estreitamento opcional** — vínculo no Cliente cobre todas as unidades salvo se restringir a específicas; menos linhas, escopo hierárquico padrão) ou exige linha **explícita por nível**? ### D-200 — PEDRA ANGULAR do isolamento: identidade é compartilhada, mas ACESSO é escopado ao TENANT ATIVO (nunca vazar entre tenants) (2026-07-11)
Alessandro ancorou o modelo na realidade e cravou o invariante de segurança mais importante. **Realidade (confirma o N:N de D-198/199):** um **médico** pode estar na iniciativa **privada** (clínica) **E** na **pública** (projeto SUS) — N tenants/clientes. Um **paciente** tem **SUS (obrigatório)** e pode ter **plano privado** — está em >1 tenant, e em >1 cliente (no mesmo tenant ou em tenants distintos). **"Tudo pode acontecer."**
**PEDRA ANGULAR (regra dura, inegociável — LGPD/saúde):** a **identidade é compartilhada** (é a MESMA pessoa, mesmo ID de sistema/golden record em todos os tenants), **MAS a VISIBILIDADE é estritamente escopada ao TENANT ATIVO**. Enquanto a pessoa opera dentro de um tenant, **só vê os dados daquele tenant** — **jamais** dado de outro tenant, **nem mesmo** de outro a que ela própria está vinculada. **Vazamento inter-tenant via identidade compartilhada é o PIOR risco** e está proibido por construção.
**Distinção que não pode ser confundida (é o que causaria o vazamento):** o **golden record (EMPI/D-191, D-198/199) resolve IDENTIDADE** ("é a mesma pessoa"); o **ACESSO é sempre escopado ao tenant ATIVO**. Um NÃO implica o outro. Refina o isolamento de D-199: **não é a UNIÃO** dos tenants vinculados que a pessoa vê — é **só o contexto ativo**. A **membership** diz em quais tenants ela **pode entrar**; o **contexto ativo** diz o que ela **vê agora** (modelo Slack/GitHub-org: 1 conta, N workspaces isolados, sem bleed).
**Implicação técnica:** o filtro de isolamento (D-197 — global query filter) é **keyed no tenant ATIVO da sessão/contexto**, não na união das memberships. **FORK/discovery (não decidir agora):** UX de **um tenant ativo por vez, troca explícita** (estilo Slack — recomendo) e como o contexto ativo é escolhido/trocado quando o Doutor logar (papel Doutor ainda não loga). Segue **depois da demo**, na fundação D-197. _(A rigidez do "1 tenant ativo/sem união" foi REFINADA em D-201 — fila unificada + visibilidade configurável.)_

### D-202 — Vaga de um doutor só é agendável a um cliente se houver VÍNCULO doutor↔cliente (gate necessário, ANTES do saldo); vínculo+vaga entram na fundação, saldo fica pra depois (2026-07-11)
Análise de use case real do Alessandro — **valida** o modelo de vínculo (D-198/199), não contradiz. **Regra (mundo real, confirmada):** um agendamento é **sempre de UM cliente, para um paciente DAQUELE cliente**. A **vaga de um doutor só aparece/é agendável para um cliente se o doutor está VINCULADO àquele cliente**. O vínculo é gate **necessário e vem ANTES do saldo** — saldo positivo **não basta**. _Ex.: Doutor A e Doutor B, ambos cardio; Piauí só tem A vinculado. B tem vagas mas **não aparece** pro Piauí, mesmo com o Piauí tendo saldo de cardio. Vincular B ao Piauí → B **aparece**._
**Dois gates distintos:** **VÍNCULO** (elegibilidade — D-198/199) × **SALDO** (orçamento `min(pool, teto)` — D-190). Vínculo primeiro.
**Consistência tri-vínculo do agendamento:** cliente-dono (D-197) + **paciente vinculado** ao cliente + **doutor vinculado** ao cliente — tudo do mesmo modelo de membership. "Colocar o doutor no cliente" = criar uma linha de vínculo.
**Resolve (confirmar na discovery) a tensão pool×dedicada (D-150×D-165, aberta em D-190):** com vínculo, "pool" deixa de ser "vaga pra todos" e vira "vaga pros clientes a que o doutor está **vinculado**". `Escala.ClienteId` (dedicada) = reserva a um cliente; sem `ClienteId` = disponível aos **vinculados** do doutor.
**SEQUÊNCIA (decisão Alessandro):** **vínculo doutor↔cliente + "vaga só aparece se vinculado" entram JUNTO com a fundação de multi-tenancy (D-197)** — jornada de escala/agendamento correta desde o início. O **saldo `min(pool, teto)` fica em FASE PRÓPRIA depois** (segue D-190): é camada separada, **não é pré-requisito** (o fluxo já funciona com vínculo + especialidade). Impacto: a **capacidade de atendimento** (quais vagas um cliente enxerga) é filtrada pelo vínculo — confirma a intuição do Alessandro de "ter um vínculo de doutor ao cliente". ### D-207 — Realidade da Unidade: local de atendimento com CAPACIDADE, ancorado em CNES (público) por cliente; reset dos agendamentos legado (2026-07-12)

- **Unidade = local de atendimento UNITÁRIO, com CAPACIDADE de atendimento** (é onde o teto/saldo vive — liga em D-190).
- **Público:** identificada por **CNES** (Cadastro Nacional de Estabelecimentos de Saúde). O CNES tem hierarquia geográfica: **CNES → município → estado → Brasil**. O **cliente público** representa um ente de governo com **tipo: municipal | estadual | federal**.
- **MESMO CNES pode existir sob VÁRIOS clientes** — ex.: Unidade A / CNES 99999 (Teresina) no cliente "Piauí" (estadual) E no cliente "Povos da Floresta" (outro ente): **mesmo CNES/nome, mas linhas Unidade DISTINTAS, IDs distintos** (donos diferentes). ⇒ **CNES é referência compartilhada, NÃO a identidade; a identidade é o Id (por cliente).** O `Codigo` único do Lote 1 precisa acomodar isso: unicidade **por-cliente** (ou a claim `unidade` referencia o Id da Unidade, não um código global).
- **Privado:** o cliente **cadastra as próprias unidades** (CNES não obrigatório).
- **Sugestão (feature futura):** ao criar cliente público, marcar o tipo → **unidades criadas por padrão** (auto-populadas do registro CNES do escopo).
**Consequência p/ o modelo (refina Lote 1, na branch — reset aceito, sem migração de dado):** `Unidade` ganha **`Cnes`** (nullable) e o `Cliente` público ganha **`TipoPublico`** (municipal/estadual/federal); a claim de login referencia a Unidade específica do cliente. A **auto-população via CNES** é feature à parte (precisa de fonte de dados CNES). **O isolamento do Lote 4 NÃO muda** (opera sobre `ClienteId`, que é derivado da Unidade). **Q2 (backfill) — decisão: ACEITAR O RESET** dos agendamentos legado (dados de demo/teste; coerente com "recomeçar do seed", D-203) — sem migração de backfill.

### D-211 — VISÃO de modelo de negócio: plataforma MULTI-EMPRESA (coopetição) + catálogo compartilhado de doutores ("Netflix/Uber de doutores"); VALIDA a fundação de multi-tenancy (2026-07-12)
Visão do Alessandro (horizonte **FUTURO/VISÃO** — desejo estratégico; valida o AGORA). A plataforma evolui de "white-label pra clientes" para uma **plataforma MULTI-EMPRESA**: outras empresas de saúde **entram como TENANTS**, a Portal dá **toda a infra de tecnologia**, e cada empresa opera seu **vertical** (Portal Telemedicina = telemedicina; outra = medicina física; etc.) — **parceiras E concorrentes** no mesmo software (coopetição). **Trunfo:** um **catálogo COMPARTILHADO de doutores — "Netflix/Uber de doutores"** — pool comum que as **unidades de atendimento** (públicas/privadas) consomem; o match é **unidade↔doutor** (não paciente↔doutor neste nível). **Efeito de rede:** quanto mais empresas/unidades, mais rico o pool = **moat**. Vínculo pode ser pra uma **empresa**, uma **unidade**, ou até pra um **CPF de médico**. Objetivo: atender **todos os públicos, em todos os espectros**.
**✅ CHECAGEM DE COERÊNCIA — isto NÃO conflita; VALIDA a fundação já construída:**
- **Empresa = nível `Tenant`** (D-199, hierarquia Tenant › Cliente › Unidade). "Multi-Tenant" deixa de ser "Fase 1 single, futuro multi" e vira **objetivo estratégico explícito**.
- **Netflix de doutores = golden record de doutor (D-198) + membership N:N (D-199)** — o doutor de plataforma vinculado a N tenants/clientes/unidades.
- **Coopetição EXIGE o isolamento estrito (D-200):** tenants concorrentes compartilham o CATÁLOGO de doutores mas **NUNCA** veem o dado um do outro. A coopetição **só é segura por causa da pedra angular "identidade compartilhada ≠ acesso"**. (Se um dia quiserem compartilhar algo entre tenants → grant explícito, D-201/204/205.)
- **Match unidade↔doutor = vaga-por-vínculo (D-202).**
- ⇒ **A arquitetura AGORA (fundação de multi-tenancy, D-197→D-208) é o enabler técnico EXATO desta Visão** — cada decisão de isolamento/vínculo que parecia "só técnica" é o que torna o "Netflix de doutores entre concorrentes" possível e seguro. A doc-da-realidade provando que o rumo está certo.
**Horizonte:** FUTURO/VISÃO. Entrada de tenants-empresa reais = onboarding + comercial, não só técnico. Reflete no farol (Visão) e no `roadmap-horizontes.md`.

### D-213 — Invariante ANTI-DOUBLE-BOOKING: uma vaga = no máximo UM agendamento ATIVO (2026-07-12)
Achado numa sondagem lógica (pedido do Alessandro "vai fazendo testes lógicos", madrugada 2026-07-13): o `POST /api/agendamentos` **não tinha guarda de unicidade por `VagaId`** — o dedup do `GET /api/vagas` (que esconde vagas já atribuídas) era a ÚNICA proteção, e ela depende do cliente/UI → vulnerável a **corrida** (2 operadores assumindo simultâneo), **lista velha** no cliente ou **duplo-clique** = a mesma vaga vendida duas vezes (pior falha de um sistema de agenda). **Correção (correção DERIVÁVEL, não regra nova — o próprio dedup do GET já intenciona "uma vaga = um agendamento", cf. o comentário "DUPLO AGENDAMENTO do mesmo slot"):** guarda no **servidor** no POST — rejeita **409 Conflict** se já existe `Agendamento` na mesma `VagaId` com `Status != Rejeitado`. **Cross-cliente** (`IgnoreQueryFilters` — a vaga é slot FÍSICO único; sem isso o filtro global por tenant esconderia a assunção de outro cliente e liberaria o duplo agendamento — mesmo motivo do dedup do GET). **Rejeitado** (cliente recusou, D-192) **NÃO** segura a vaga → permite reassumir. Cercado por 2 testes (`AntiDoubleBookingTests`: 2º POST na mesma vaga → 409 + só 1 no banco; vaga com agendamento Rejeitado → reassumível). **Follow-up:** hardening por **índice único PARCIAL** no banco (`VagaId where Status<>Rejeitado`) fecha a corrida TOCTOU concorrente-real (a guarda de app cobre os casos realistas — stale/duplo-clique/operadores em sequência). **Nuance a confirmar com o humano:** quais status seguram a vaga (hoje: todos menos Rejeitado — Pendente/Enviando/Confirmado/FalhaTemp seguram). Relaciona: D-192, D-184 (guard de escala×agendamento). **→ RESOLVIDA em D-214** (a nuance foi decidida: TODOS os status seguram — derivado de D-192).

### D-214 — POST /agendamentos valida a vaga REAL (isolamento na ESCRITA) + Rejeitado também segura a vaga (2026-07-12)
Continuação da sondagem lógica (madrugada 2026-07-13). Uma auditoria adversarial achou um **cluster crítico** no `POST /api/agendamentos`, todo com a mesma raiz — **o POST não resolvia/validava a vaga, confiava no body**:
- **(Achado 1, crítico) Vaga FORJADA:** aceitava qualquer `VagaId` string — dava pra agendar um slot de **dia de folga, fora de vigência, médico inativo, bloco inexistente** → agendamento-fantasma que o `GET /api/vagas` nunca ofertou (infla capacidade, trava editar/excluir a escala via D-184, gera outbox de sync fantasma).
- **(Achado 2, crítico — multitenancy/LGPD) Isolamento só na LEITURA:** o POST validava a unidade contra o token mas **não reaplicava o filtro de vínculo (D-202/D-203)** → um **Operador do cliente A conseguia agendar a vaga de um doutor dedicado/vinculado ao cliente B**, consumindo capacidade de B e carimbando como A. A isolação vaga-por-vínculo do Lote 5 valia só no `GET`.
- **(Achado 4, dinheiro) Campos forjáveis:** `Especialidade`/`MedicoNome` gravavam crus do body → atribuição de **faturamento** (D-169, por médico×especialidade) forjável.
**Correção (correção DERIVÁVEL — a lógica de oferta já existia):** extraiu-se a derivação do `GET /api/vagas` num helper único `EscalaEndpoints.DerivarVagasOfertadasAsync(db, tenantEscopo, ct)` (fonte da verdade das vagas ofertadas ao escopo). O `POST` agora **valida pertencimento**: só aceita um `VagaId` que o GET do MESMO escopo realmente ofertaria — senão **409**. Isso fecha os 3 achados de uma vez (forjada, cross-cliente, e usa `Especialidade`/`MedicoNome` da **vaga resolvida**). É *estritamente mais restritivo* (só rejeita mais, nunca abre brecha). Cercado por `AgendamentoVagaRealTests` (happy-path preserva o agendar; vaga de domingo forjada → 409; operador de C1 agendando vaga dedicada a C2 → 409). 11 testes pré-existentes que usavam `VagaId` forjado (o próprio furo) foram adaptados a semear **vaga real** (revisor-seguranca conferiu: nenhuma asserção enfraquecida). GET idêntico.
**Refina a nuance aberta da D-213 (Rejeitado):** a guarda anti-double-booking passou a bloquear a vaga por **QUALQUER status** (antes: todos menos Rejeitado). **Derivação de D-192:** `Rejeitado` = "o cliente diz que a vaga já está ocupada" = **indisponível** → não deve ser re-agendável; e é coerente com o `GET`, que já esconde qualquer vaga com agendamento (rejeitado inclusive). Reabrir uma vaga rejeitada = **feature futura** com sinal explícito do cliente (não é o default). ⚠️ **Aberto pro Alessandro reverter:** se a intenção for "Rejeitado LIBERA a vaga", é só dizer — hoje é seguro/neutro (a integração real está DEP-TC-1, nenhum agendamento vira Rejeitado ainda). **Follow-ups (não-bloqueantes, revisor-seguranca):** (a) índice único parcial no banco p/ a corrida TOCTOU; (b) `clienteId`-da-unidade × `tenantEscopo.ClienteId`-do-claim podem divergir com token inconsistente (não explorável pelos papéis atuais); (c) `PrimeiraSegunda()` de alguns testes pode cair no mês seguinte nos últimos dias do mês → flaky. Relaciona: D-192, D-202/D-203 (vaga-por-vínculo), D-206 (fail-closed), D-213.

### D-216 — Modelo de domínio da UNIDADE: TIPO + PREFERENCIAL DE ATENDIMENTO (CNES só p/ tipo saúde) (2026-07-13)
Direção do Alessandro (refinando D-215). **"A base do CNES é uma coisa; o vínculo dele com a Unidade é outra"** — nem toda Unidade tem CNES. A **Unidade** ganha atributos próprios:
- **Tipo** (categoria de onde o atendimento acontece): (1) **Estabelecimento de saúde** — tem **CNES** → logo tem **endereço** (vem da base pública D-215); (2) **Privado**; (3) **Escola**; (4) **Unidade prisional**; (5) **Outros**.
- **Preferencial de atendimento**: **Presencial** | **Remoto** — define o **DEFAULT da modalidade no agendamento**. "Se o tipo é saúde e presencial, os atendimentos ali nascem presenciais; o mesmo pro remoto."
- **CNES/endereço só importam pro tipo com estabelecimento físico.** Existem **unidades de saúde VIRTUAIS** (ex.: ambulatório virtual) — **endereço não importa, o atendimento é no app** (remoto). Essas não dependem de CNES.
**Perguntas abertas — RESOLVIDAS pelo Alessandro ("perfeito, isso reflete a realidade", 2026-07-13):**
(a) ✅ **Tipo e Natureza são ORTOGONAIS** — `Tipo` = contexto (`SAÚDE · ESCOLA · PRISIONAL · OUTROS`) × `Natureza` = (`PÚBLICO | PRIVADO`). Clínica privada = tipo Saúde + natureza Privado (com CNES). Some a sobreposição.
(b) ✅ **Preferencial = DEFAULT overridável** — o agendamento NASCE com a modalidade da unidade (presencial/remoto), mas pode mudar caso a caso (não trava).
(c) ✅ **"Virtual" = ATRIBUTO, não tipo** — flag `virtual`/`semEndereço` numa unidade de saúde (dispensa CNES/endereço, atendimento no app), não um 6º tipo.
(d) ✅ **Unidade↔Cliente = vínculo EXPLÍCITO/curado** — a unidade nasce da base pública; quem é de qual cliente é definido pelo admin/cliente (espelha o vínculo de doutor, D-197).
Relaciona: D-215 (base CNES), D-207 (Unidade/CNES), D-197. Spec: `specs/multi-tenancy/cnes-e-unidade.md`.
